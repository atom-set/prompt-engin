## 一、素材类型

### 1.1 Chat 沟通记录 ⭐ **最有价值**

**适用场景**：
- 展示 AI 辅助开发的实际对话过程
- 证明 Prompt Engine 规则的实际效果
- 展示问题解决流程

**收集内容**：
- ✅ **问题发现**：发现提示词漏洞或问题的对话
- ✅ **问题修复**：修复提示词漏洞的对话过程
- ✅ **同步过程**：同步提示词到多个项目的对话
- ✅ **效果验证**：验证修复效果的对话
- ✅ **使用场景**：实际使用 Prompt Engine 的对话

**示例场景**：
```
场景1：发现提示词漏洞
用户：按住点击 wx_login_redirect 不会进入定义的文件
AI：分析问题...输出方案...执行修复
结果：问题解决，需要同步到其他项目

场景2：同步提示词
用户：需要同步到项目B、C、D
AI：使用同步脚本...批量同步完成
结果：所有项目同步成功
```

### 1.2 数据对比素材

**适用场景**：
- 展示使用前后的对比效果
- 证明实际收益

**收集内容**：
- ✅ **时间对比**：维护时间从 X 小时 → Y 分钟
- ✅ **成本对比**：Token 占用减少的具体数值
- ✅ **效率对比**：同步效率提升的具体案例
- ✅ **版本对比**：版本一致性改善的具体数据

**示例**：
```
使用前：
- 维护时间：2 小时（手动同步 20 个项目）
- Token 占用：8597 行
- 版本一致性：5 个项目版本不一致

使用后：
- 维护时间：5 分钟（一键同步）
- Token 占用：3427 行
- 版本一致性：所有项目版本统一
```

### 1.3 实际案例素材

**适用场景**：
- 展示不同场景下的实际应用
- 证明方案的通用性

**收集内容**：
- ✅ **项目背景**：项目类型、规模、团队情况
- ✅ **遇到的问题**：具体的问题描述
- ✅ **解决方案**：使用的 Prompt Engine 方案
- ✅ **实际效果**：具体的效果数据或反馈

**示例格式**：
```markdown
## 案例：XX 项目

**项目背景**：
- 项目类型：前端项目
- 团队规模：10 人
- 项目数量：20 个

**遇到的问题**：
- 提示词分散在 20 个项目中
- 发现漏洞后需要手动同步
- 维护成本极高

**解决方案**：
- 使用方式2（精简版 + 技能系统）
- 集中管理提示词
- 批量同步工具

**实际效果**：
- 维护时间：从 2 小时 → 5 分钟
- Token 占用：减少 60%
- 版本一致性：100%
```

### 1.4 截图和演示素材

**适用场景**：
- 可视化展示效果
- 增强说服力

**收集内容**：
- ✅ **对话截图**：Chat 沟通记录的关键截图
- ✅ **工具使用截图**：同步脚本执行截图
- ✅ **效果对比截图**：使用前后的对比
- ✅ **数据统计截图**：统计脚本的输出

## 二、素材收集方法

### 2.1 Chat 沟通记录收集

**方法1：导出对话记录**

如果使用 Cursor、TRAE、Antigravity 等工具：
- 查找相关的对话记录
- 导出或复制关键对话片段
- 整理成 Markdown 格式

**方法2：手动整理**

- 回顾使用 Prompt Engine 的关键对话
- 记录问题、解决方案、效果
- 整理成案例格式

### 2.2 数据统计收集

**方法1：使用统计脚本**

```bash
# 运行统计脚本
bash scripts/utils/stat_metrics.sh

# 记录输出结果
```

**方法2：手动统计**

- 记录使用前后的数据
- 对比分析
- 整理成表格

### 2.3 案例素材收集

**收集流程**：
1. 回顾使用 Prompt Engine 的项目
2. 记录项目背景、问题、解决方案、效果
3. 整理成标准案例格式

## 四、素材使用建议

### 4.1 在分享中的使用

#### 使用位置1：实际案例部分（五、使用场景）

**建议**：
- 使用真实的 Chat 沟通记录
- 展示完整的对话过程
- 突出关键问题和解决方案

**示例**：
```markdown
### 5.2 实际案例

**案例1：提示词漏洞修复和同步**

**场景**：发现提示词漏洞，需要修复并同步到多个项目

**对话记录**：
[使用 Chat 沟通记录]

**效果**：
- 问题快速定位和修复
- 一键同步到所有项目
- 维护时间从 2 小时 → 5 分钟
```

#### 使用位置2：数据对比部分（三、核心价值）

**建议**：
- 使用实际统计数据
- 展示使用前后的对比
- 突出实际收益

**示例**：
```markdown
### 3.2 实际效果

**实际案例数据**（来自 XX 项目）：
- 维护时间：从 2 小时 → 5 分钟
- Token 占用：从 8597 行 → 3427 行
- 版本一致性：从 60% → 100%
```

### 4.2 素材选择原则

**优先选择**：
- ✅ 最能体现核心价值的案例
- ✅ 数据最完整的案例
- ✅ 最有代表性的场景

**避免使用**：
- ❌ 过于复杂的案例（难以理解）
- ❌ 数据不完整的案例
- ❌ 过于特殊的场景（缺乏代表性）

### 4.3 素材处理建议

**隐私处理**：
- 去除敏感信息（项目名称、代码细节等）
- 使用通用化表述
- 保留核心问题和解决方案

**内容精简**：
- 提取关键对话片段
- 去除冗余信息
- 突出核心价值点

**格式统一**：
- 使用统一的格式模板
- 保持风格一致
- 便于理解和展示

## 五、素材提交方式

### 5.1 提交格式

**方式1：直接提供 Chat 记录**
- 复制对话记录
- 整理成 Markdown 格式
- 提交到 `materials/chat-records/` 目录

**方式2：提供案例描述**
- 描述项目背景、问题、解决方案、效果
- 整理成案例格式
- 提交到 `materials/cases/` 目录

**方式3：提供数据对比**
- 提供使用前后的数据
- 整理成对比表格
- 提交到 `materials/data-comparison/` 目录

### 5.2 提交要求

- ✅ 提供项目背景信息（类型、规模等）
- ✅ 说明遇到的问题
- ✅ 描述使用的解决方案
- ✅ 提供实际效果数据或反馈
- ✅ 标注数据来源（已统计/估算/定性分析）
